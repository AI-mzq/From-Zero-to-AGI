# 07_视频多模态

> 视频多模态是在图像基础上引入时间维度：模型不仅要看清每一帧，还要理解动作、事件、因果、节奏、声音和跨帧一致性。

**By：猫先生 of 「魔方AI空间」**

## 本章导读

前面两章分别讲了 [视觉语言模型 VLM](../05_视觉语言模型VLM/README.md) 和 [多模态生成模型](../06_多模态生成模型/README.md)。

视频多模态站在这两者中间：

```text
视频理解：
  视频 -> 内容描述 / 问答 / 摘要 / 事件识别 / 时序定位

视频生成：
  文本 / 图像 / 音频 / 控制条件 -> 视频
```

视频比图像多了时间维度，也就多了很多长期问题：

- 如何抽帧？
- 如何建模动作和事件？
- 如何处理长视频？
- 如何压缩视觉 Token？
- 如何融合声音、字幕和画面？
- 如何保持生成视频的主体、场景和运动一致性？
- 如何评测一个视频模型真的理解了时间，而不是只看到了几张关键帧？

本章不追逐某个短期模型榜单，而是建立一套几年后仍然有用的视频多模态框架。

## 一句话理解视频多模态

视频多模态可以理解为：

> 让模型在时间轴上理解和生成视觉、语言、音频等多种信息。

图像回答的是“这一刻看到了什么”，视频还要回答：

- 之前发生了什么？
- 现在正在发生什么？
- 接下来可能发生什么？
- 哪个动作导致了哪个结果？
- 声音和画面是否对应？
- 这个事件发生在第几秒？
- 视频整体想表达什么？

所以视频不是多张图片的简单堆叠。它是带时间结构的多模态序列。

## 视频为什么比图像更难？

视频比图像难，核心原因不是“数据更多”这么简单，而是信息结构更复杂。

| 难点 | 说明 |
| --- | --- |
| 时间维度 | 模型要理解动作、变化、顺序和持续时间 |
| Token 成本 | 多帧输入会快速消耗视觉 Token 和上下文窗口 |
| 长程依赖 | 关键事件可能分散在很远的时间点 |
| 主体一致性 | 同一个人、物体或场景要跨帧保持身份 |
| 多模态融合 | 画面、语音、字幕、背景音可能共同决定语义 |
| 时序定位 | 问题常常要求定位到具体片段或时间范围 |
| 运动建模 | 生成视频时要保证运动自然、物理合理 |
| 评测困难 | 视频任务没有单一标准答案，人工评估成本高 |

一个模型能看懂单张图，不代表它能理解视频。视频理解的关键，是让模型看到变化，而不只是看到画面。

## 视频多模态的两条主线

视频多模态可以分成两条主线。

| 主线 | 目标 | 典型任务 |
| --- | --- | --- |
| 视频理解 | 从视频中提取语义、事件、动作和关系 | 视频问答、Caption、摘要、时序定位、异常检测 |
| 视频生成 | 根据条件生成或编辑视频 | 文生视频、图生视频、视频编辑、动作控制、风格迁移 |

两条线正在融合。

例如，一个视频编辑系统可能需要：

```text
先理解原视频中人物、动作和镜头
  -> 再根据用户指令修改某个片段
  -> 保持主体和运动一致
```

理解是生成的前提，生成也会反过来推动理解能力的评测。

## 视频输入由什么组成？

视频不是只有帧。

一个真实视频样本通常包含：

| 组成 | 说明 |
| --- | --- |
| 视频帧 | 画面内容，是视觉信息主体 |
| 时间戳 | 每帧或片段的位置 |
| 音频 | 语音、音乐、环境声、事件声 |
| 字幕 | 人声转写、翻译字幕、硬字幕 |
| 镜头结构 | 转场、切镜、场景变化 |
| 元数据 | 标题、描述、标签、上传时间等 |
| 用户问题 | 决定模型需要关注什么 |

很多视频任务不能只靠画面完成。

例如：

- 问“这个人说了什么”，需要语音或字幕。
- 问“爆炸发生前有什么声音”，需要音频。
- 问“第二个镜头的地点在哪里”，需要镜头切分。
- 问“视频表达的主要观点是什么”，需要跨片段整合。

所以视频多模态天然是视觉、语言、音频和时间结构的融合问题。

## 视频理解的基本流程

一个常见视频理解系统可以简化为：

```text
视频
  -> 抽帧 / 切片 / 镜头分割
  -> 视频编码器 / 图像编码器
  -> 时间建模模块
  -> 连接器
  -> LLM
  -> 回答 / 摘要 / 定位 / 结构化输出
```

如果加入音频和字幕：

```text
视频帧 + 音频 + 字幕 + 用户问题
  -> 多模态编码
  -> 时序对齐
  -> LLM 推理
  -> 输出答案
```

这里最难的部分往往不是某一个模块，而是三个约束同时存在：

- 输入很长。
- 用户问题很开放。
- 视觉、语音和文本需要在时间轴上对齐。

## 抽帧：视频多模态的第一道取舍

视频模型通常不可能把所有帧都完整送进 LLM。

例如一个 10 分钟、30 FPS 的视频有：

```text
10 分钟 x 60 秒 x 30 帧 = 18000 帧
```

如果每帧都当作一张图处理，成本会非常高。

因此抽帧策略非常关键。

| 抽帧方式 | 优点 | 局限 |
| --- | --- | --- |
| 均匀抽帧 | 简单稳定，覆盖整体时间 | 可能错过短暂关键事件 |
| 关键帧抽取 | 更关注画面变化 | 依赖镜头变化或启发式规则 |
| 事件驱动采样 | 更贴近任务语义 | 需要额外检测或预处理 |
| 问题感知采样 | 根据用户问题选择片段 | 实现复杂，对检索和定位要求高 |
| 多尺度采样 | 同时看全局和局部 | Token 预算和调度更复杂 |

抽帧不是工程细节，而是视频理解能力的一部分。

如果关键帧没被采到，后面的模型再强也很难答对。

## 视觉 Token 预算

视频输入最容易遇到 Token 爆炸。

假设每帧保留 256 个视觉 Token：

```text
8 帧：2048 视觉 Token
32 帧：8192 视觉 Token
128 帧：32768 视觉 Token
```

这还没有算用户问题、字幕、历史对话和模型输出。

所以视频多模态必须处理视觉预算问题：

- 每段视频选多少帧？
- 每帧保留多少视觉 Token？
- 是否压缩成片段级 Token？
- 是否只保留和问题相关的片段？
- 是否用分层记忆保存长视频信息？

长期来看，视频模型的竞争不只是“看更多帧”，而是“在有限预算里看对内容”。

## 时序建模的层级

视频的时间结构可以按不同粒度建模。

| 粒度 | 对象 | 典型问题 |
| --- | --- | --- |
| 帧级 | 单帧画面 | 这一帧有什么？ |
| 短片段级 | 几帧到数秒 | 发生了什么动作？ |
| 镜头级 | 一个连续镜头 | 这个镜头表达什么？ |
| 事件级 | 一个完整事件 | 事件如何开始、发展和结束？ |
| 故事级 | 整个视频 | 视频主旨、叙事和观点是什么？ |

不同任务需要不同粒度。

问“杯子是什么颜色”可能只需要一帧；问“他为什么摔倒”则需要前后动作和环境线索；问“一段会议视频的结论是什么”，需要长程语义整合。

## 视频编码器的演进

视频编码器的发展，可以粗略分成几类。

| 路线 | 代表思想 | 适合理解 |
| --- | --- | --- |
| 2D CNN + 时间聚合 | 每帧提特征，再做时序融合 | 把视频当图像序列 |
| Two-Stream | 分别处理 RGB 和光流 | 显式建模外观与运动 |
| 3D CNN | 在空间和时间上一起卷积 | 直接学习局部时空模式 |
| Non-local / Attention | 建模远距离时空关系 | 捕捉长程依赖 |
| Video Transformer | 用 Transformer 处理时空 Token | 统一建模空间和时间 |
| 自监督视频预训练 | 从大量视频中学习通用表示 | 降低标注依赖 |
| Video-LLM | 视频特征接入 LLM | 开放式问答和推理 |

这条演进线背后有一个长期问题：视频模型如何同时处理局部运动、全局语义和长程时间关系。

## 从图像编码器到视频编码器

很多现代视频多模态系统并不一定从零训练视频编码器，而是复用强图像编码器。

常见做法是：

```text
视频 -> 抽帧
每帧 -> 图像编码器
帧特征 -> 时间聚合 / 连接器 / LLM
```

这样做的优势：

- 可以复用成熟的视觉编码器。
- 工程成本低。
- 适合短视频、稀疏事件和图文问答式任务。

局限：

- 对细微运动不敏感。
- 对动作连续性建模不足。
- 容易把视频理解退化成“多图理解”。

真正的视频理解需要时序建模，而不是只把若干帧拼给 LLM。

## Video-LLM 的基本结构

Video-LLM 通常把视频特征接入语言模型，让 LLM 负责开放式问答、摘要和推理。

一个典型结构是：

```text
视频
  -> 抽帧 / 切片
  -> 视觉或视频编码器
  -> 时间聚合模块
  -> 连接器
  -> LLM
  -> 视频问答 / 摘要 / 推理
```

它和图像 VLM 的区别在于：中间多了时间维度。

| 模块 | 图像 VLM | Video-LLM |
| --- | --- | --- |
| 输入 | 单图或多图 | 多帧、片段、音频、字幕 |
| 编码 | 视觉编码器 | 图像编码器 / 视频编码器 |
| 连接 | Projector、Q-Former 等 | 还需要时间聚合和片段压缩 |
| 上下文 | 视觉 Token + 文本 | 时序视觉 Token + 字幕 + 音频 + 文本 |
| 输出 | 图文问答 | 视频问答、摘要、事件定位、动作推理 |

Video-LLaMA、Video-ChatGPT、Video-LLaVA 等工作都体现了这条路线。

## 视频问答

视频问答要求模型基于视频内容回答问题。

问题类型可以非常不同：

| 问题类型 | 示例 |
| --- | --- |
| 感知类 | 视频里出现了哪些物体？ |
| 动作类 | 人物在做什么？ |
| 顺序类 | 他先做了什么，后做了什么？ |
| 因果类 | 为什么杯子掉到了地上？ |
| 计数类 | 球被传了几次？ |
| 时间定位 | 事故发生在视频的哪个阶段？ |
| 多模态融合 | 画面中发生响声时，人物在做什么？ |

真正有价值的视频问答，往往不是问“画面中有什么”，而是问“变化意味着什么”。

## 视频 Caption 与摘要

视频 Caption 要求模型描述视频内容。

短视频 Caption 关注动作和场景：

```text
一名男子走进厨房，拿起杯子并倒了一杯水。
```

长视频摘要则更关注结构：

```text
视频先介绍问题背景，随后展示实验步骤，最后总结主要结论。
```

Caption 和摘要的难点不同：

- Caption 要精确描述局部动作。
- 摘要要压缩长程信息。
- 教程视频要保留步骤。
- 会议视频要保留观点和结论。
- 监控视频要保留异常和时间点。

所以视频摘要不是简单把每帧 caption 拼起来，而是要做时序组织和信息压缩。

## 时序定位

时序定位要求模型找到事件发生的时间范围。

例如：

```text
问题：车门是在什么时候打开的？
输出：大约在 00:12 到 00:15 之间。
```

这类任务对视频系统很重要，因为用户经常不只想知道答案，还想知道证据在哪里。

时序定位需要模型具备：

- 对事件的语义理解。
- 对时间轴的对齐能力。
- 对短暂动作的敏感度。
- 对答案证据的引用能力。

在真实应用中，带时间戳的答案比纯文本答案更有用。

## 长视频理解

长视频理解是视频多模态最难的问题之一。

长视频可能包括：

- 课程录像
- 会议录屏
- 电影和纪录片
- 直播回放
- 监控视频
- 用户操作录像
- 机器人第一视角视频

长视频的难点在于：

```text
输入极长
  -> 不能全部塞进上下文
  -> 必须分段、检索、压缩和记忆
```

常见思路包括：

| 方法 | 作用 |
| --- | --- |
| 分段摘要 | 把长视频切成片段，逐段生成摘要 |
| 层级记忆 | 帧级、片段级、视频级多层表示 |
| 视频检索 | 根据问题找相关片段 |
| 时间索引 | 为事件建立可检索时间戳 |
| 多轮聚合 | 先局部理解，再全局归纳 |
| 字幕辅助 | 用语音转写降低视觉负担 |

长视频理解的长期方向，不是把上下文无限变长，而是让模型学会建立可查询的视频记忆。

## 音频、字幕和画面融合

视频多模态经常需要同时处理画面和声音。

音频能提供画面看不到的信息：

- 人在说什么。
- 是否有敲门声、警报声、碰撞声。
- 音乐情绪和节奏。
- 环境是否嘈杂。
- 事件发生时是否有声音提示。

字幕和语音转写也很关键：

- 对课程、访谈、会议、新闻尤其重要。
- 能显著降低纯视觉理解压力。
- 但也可能有转写错误或缺失。

可靠的视频系统应该能区分：

```text
画面证据
语音证据
字幕证据
模型推断
```

这对减少幻觉和提高可解释性很重要。

## 视频生成的基本问题

视频生成是在时间轴上生成视觉内容。

相比图像生成，它多了三个核心约束：

| 约束 | 说明 |
| --- | --- |
| 时间一致性 | 主体、场景、风格不能每帧漂移 |
| 运动合理性 | 动作、物理关系和镜头运动要自然 |
| 长程结构 | 多秒甚至多分钟内容要有节奏和叙事 |

一个简化的视频生成流程是：

```text
文本 / 图像 / 控制条件
  -> 视频生成模型
  -> 多帧序列
  -> 后处理 / 插帧 / 超分 / 音频
  -> 视频输出
```

视频生成不是把图像生成模型循环调用多次。逐帧生成很容易导致闪烁、漂移和主体变形。

## 文生视频

文生视频根据文本描述生成视频。

它要同时解决：

- 文本语义对齐。
- 场景构建。
- 镜头运动。
- 主体动作。
- 物理连续性。
- 风格一致性。
- 时间长度控制。

例如：

```text
Prompt：一只橙色小猫在雨后的街道上跳过水坑，镜头缓慢跟随。
```

模型不仅要生成猫和街道，还要生成跳跃动作、雨后环境、镜头跟随和跨帧一致的主体。

## 图生视频与首帧控制

图生视频以图像作为起点生成后续运动。

它比纯文生视频更可控，因为首帧提供了明确的主体、构图和风格。

典型任务包括：

- 让一张人物图动起来。
- 根据产品图生成展示视频。
- 根据插画生成动态短片。
- 根据首帧生成镜头推进。

但图生视频也有难点：

- 运动不能破坏主体身份。
- 后续帧要延续首帧构图和风格。
- 动作幅度过大时容易变形。
- 镜头变化需要和首帧透视一致。

## 视频编辑

视频编辑比视频生成更接近生产工作流。

常见任务包括：

- 替换背景
- 修改人物服装
- 改变天气或时间
- 删除物体
- 局部重绘
- 风格化
- 延长视频
- 改变动作或镜头

视频编辑的核心问题是：

```text
该改的要改
不该改的要保持
跨帧还要一致
```

图像编辑只需要保留单张图的一致性，视频编辑还要保留时间一致性。

这也是为什么视频编辑通常比视频生成更难落地：用户对“保持原视频”的要求非常高。

## 运动控制

视频生成的可控性很大程度上取决于运动控制。

常见控制信号包括：

| 控制信号 | 作用 |
| --- | --- |
| 文本动作描述 | 指定大致动作 |
| 姿态序列 | 控制人物骨架运动 |
| 深度序列 | 控制空间结构 |
| 光流 | 控制像素运动趋势 |
| 相机轨迹 | 控制镜头运动 |
| 分割 Mask | 控制区域或主体 |
| 关键帧 | 控制重要时间点 |
| 音频节奏 | 控制口型、舞蹈或剪辑节奏 |

长期来看，视频生成会越来越像“可控导演系统”，而不是一次性抽样工具。

## 视频生成的评测

视频生成评测比图像更难，因为它要看时间。

| 维度 | 核心问题 |
| --- | --- |
| 视觉质量 | 单帧是否清晰、自然、有细节？ |
| 文本对齐 | 是否符合 Prompt？ |
| 时间一致性 | 主体、场景、风格是否跨帧稳定？ |
| 运动质量 | 动作是否自然、物理是否合理？ |
| 镜头语言 | 镜头运动、景别和节奏是否合理？ |
| 长程结构 | 多秒内容是否有连续事件和叙事？ |
| 可控性 | 姿态、首帧、关键帧等条件是否生效？ |
| 编辑保真 | 未编辑区域是否保持稳定？ |
| 音画同步 | 口型、声音、动作是否一致？ |
| 安全性 | 是否生成有害、侵权或误导性内容？ |

只看单帧截图会高估视频生成能力。视频模型必须在播放中评估。

## 视频理解的评测

视频理解评测也不能只看选择题分数。

更完整的评测应包括：

| 维度 | 核心问题 |
| --- | --- |
| 事件理解 | 是否理解发生了什么？ |
| 时间顺序 | 是否知道先后关系？ |
| 动作识别 | 是否能识别动作和状态变化？ |
| 时序定位 | 是否能指出事件发生时间？ |
| 长视频摘要 | 是否保留关键结构和结论？ |
| 多模态融合 | 是否结合画面、音频和字幕？ |
| 视觉细节 | 是否识别小物体、文字和局部变化？ |
| 因果推理 | 是否理解动作导致的结果？ |
| 幻觉控制 | 是否编造不存在的事件？ |

评测一个视频模型，最好同时包含短视频、长视频、带字幕视频、无字幕视频、动作密集视频和静态讲解类视频。

## 视频多模态的常见失败模式

### 1. 把视频当成图片集合

模型只描述几个关键帧，没有真正理解动作、变化和时间顺序。

### 2. 错过短暂关键事件

抽帧策略没采到关键帧，模型自然无法回答。

### 3. 前后顺序颠倒

模型看到两个事件，但把先后关系说反。

### 4. 长视频遗忘

早期片段的信息在后续问答中丢失，或者摘要只保留最后几分钟内容。

### 5. 音画不同步

模型只看画面，没有利用语音或音效，导致回答不完整。

### 6. 生成视频主体漂移

同一个人物、角色、产品在不同帧变样。

### 7. 运动不自然

动作突然跳变、物体穿模、镜头漂移、物理关系不合理。

### 8. 编辑越界

用户只想修改局部区域，模型却改变了整个场景或主体身份。

## 视频多模态和 Agent

视频多模态会成为多模态 Agent 的重要感知来源。

例如：

```text
屏幕录制
  -> 识别用户操作过程
  -> 总结错误步骤
  -> 给出修复建议

机器人第一视角视频
  -> 理解环境变化
  -> 规划下一步动作
  -> 执行并观察反馈

监控视频
  -> 检测异常事件
  -> 定位时间点
  -> 生成结构化报告
```

在这些场景里，模型需要的不只是“看懂视频”，还要能把视频理解结果变成行动依据。

## 工程视角：视频系统怎么搭

真实视频系统通常不是单个模型端到端解决。

更常见的是组合式架构：

```text
视频切分
  -> 抽帧 / 音频转写 / 镜头检测
  -> 视频特征索引
  -> 相关片段检索
  -> VLM / Video-LLM 分析
  -> LLM 汇总和推理
  -> 时间戳引用 / 报告 / 操作建议
```

生成系统也类似：

```text
意图理解
  -> 分镜规划
  -> 关键帧生成
  -> 视频生成
  -> 一致性修复
  -> 超分 / 插帧 / 配音 / 剪辑
  -> 安全审核和导出
```

长期来看，视频能力不是只靠一个“大模型按钮”。采样、检索、索引、生成、编辑、审核和交互共同决定系统质量。

## 如何阅读视频多模态论文？

读视频多模态论文，可以按下面几个问题拆解：

1. **任务是什么？** 视频理解、视频问答、时序定位、文生视频、图生视频，还是视频编辑？
2. **输入如何表示？** 帧、片段、光流、音频、字幕、时间戳是否参与？
3. **如何处理时间？** 2D 帧聚合、3D 卷积、Video Transformer、时序记忆还是检索？
4. **Token 预算怎么控制？** 抽多少帧、每帧多少 Token、是否做片段压缩？
5. **是否支持长视频？** 只是短 clip，还是能处理分钟级、小时级内容？
6. **是否利用音频和字幕？** 还是只看画面？
7. **是否有时序定位能力？** 能否给出证据时间戳？
8. **生成是否可控？** 文本、首帧、姿态、关键帧、相机运动是否可控？
9. **评测是否覆盖时间问题？** 是否只测静态语义，还是测试顺序、因果和长程依赖？
10. **失败案例是什么？** 是否分析漏采样、时间错乱、主体漂移和运动异常？

这套问题比记住具体模型名字更耐用。

## 长期不变的判断框架

视频多模态发展很快，但下面这些问题会长期存在。

### 1. 是否真的理解时间？

模型能不能判断先后、持续、转折、因果和变化，是视频能力的核心。

### 2. 是否在有限预算里看对内容？

视频太长，不可能全看。抽帧、检索、压缩和记忆决定模型是否抓住关键片段。

### 3. 是否能跨模态对齐？

画面、声音、字幕和用户问题需要在时间轴上对齐。只看单一模态容易漏信息。

### 4. 是否能给出证据位置？

时间戳引用会让视频理解结果更可信，也更适合真实应用。

### 5. 是否能保持一致性？

生成视频必须保持人物、物体、风格、运动和镜头的连续性。

### 6. 是否支持可控编辑？

真实工作流需要局部修改、关键帧控制、动作控制和版本迭代。

### 7. 是否能进入系统闭环？

视频模型最终要进入检索、剪辑、监控、教学、Agent、创作和生产系统，而不是只展示单个样例。

## 常见误区

### 1. 视频理解不是多图问答

多图问答可以处理若干帧，但视频理解必须处理时间关系。

### 2. 看更多帧不一定更好

如果采样和压缩策略不好，更多帧只会增加成本和噪声。

### 3. 高分辨率不能替代时间建模

图像看得再清楚，也不能自动理解动作顺序和事件因果。

### 4. 视频生成不是逐帧生成图片

逐帧生成容易闪烁和漂移。视频生成必须显式处理时间一致性。

### 5. 短视频能力不等于长视频能力

几秒 clip 的问答能力，不能直接外推到会议、课程、电影或监控长视频。

### 6. 没有时间戳的答案在很多场景不够用

教程、监控、会议、体育分析等任务都需要知道证据发生在什么时候。

## 本章小结

视频多模态的核心可以概括为：

```text
图像理解
  + 时间建模
  + 音频 / 字幕融合
  + Token 预算控制
  + 长视频记忆
  + 一致性生成
  + 可控编辑
```

几年之后，具体模型和产品会变化，但这些问题仍然会在：

- 视频问答
- 长视频摘要
- 视频检索
- 视频 Agent
- 文生视频
- 图生视频
- 视频编辑
- 影视和内容生产
- 监控、教育、工业和机器人场景

中反复出现。

学习视频多模态，最重要的是抓住一句话：视频不是图片集合，而是带时间结构的多模态世界。
