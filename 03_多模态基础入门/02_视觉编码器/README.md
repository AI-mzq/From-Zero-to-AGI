# 02_视觉编码器

> 视觉编码器是多模态模型的“眼睛”。它负责把图像、截图、文档页或视频帧转换成模型能处理的视觉特征，再交给连接器和 LLM 使用。

**By：猫先生 of 「魔方AI空间」**

## 本章导读

在多模态模型里，LLM 本身并不会直接“看见”图片。图片首先要经过视觉编码器，变成一串视觉特征或视觉 Token。

一个常见流程是：

```text
图像
  -> 切分 / 预处理
  -> 视觉编码器
  -> 视觉特征 / 视觉 Token
  -> 连接器
  -> LLM
```

这一章重点回答：

- 图像为什么不能直接输入 LLM？
- CNN、ViT、Swin Transformer 的视觉建模思路有什么不同？
- Patch Embedding 和视觉 Token 是什么？
- CLIP、SigLIP、DINOv2 这类视觉编码器为什么常被 MLLM 使用？
- 分辨率、Patch 大小、视觉 Token 数如何影响效果和成本？
- 视频帧、文档截图、GUI 截图对视觉编码器有什么额外要求？

## 一句话理解视觉编码器

视觉编码器可以理解为：

> 把像素空间里的图像，压缩成一组带有语义和空间信息的向量表示。

这组向量后续可以用于：

- 图像分类
- 图文检索
- 图像描述
- 视觉问答
- OCR 和文档理解
- 视频理解
- GUI Agent
- 多模态 RAG

视觉编码器越好，模型越容易从图像中提取可靠信息；但它不是全部。连接器、LLM、训练数据和评测同样重要。

## 为什么需要视觉编码器？

文本可以被 Tokenizer 切成离散 Token，例如：

```text
“一只猫坐在桌子上”
  -> Token 1, Token 2, Token 3, ...
```

图像则是像素矩阵，例如一张 224 x 224 的 RGB 图片，本质上是：

```text
224 x 224 x 3 个数值
```

如果直接把像素交给 LLM，至少会遇到三个问题：

| 问题 | 说明 |
| --- | --- |
| 数据形态不同 | LLM 处理 Token 序列，图像是二维像素网格 |
| 信息量过大 | 原始像素太多，直接进入 Transformer 成本很高 |
| 语义层级不同 | 像素本身不是“猫”“桌子”“文字”这样的语义单位 |

视觉编码器的作用，就是把低层像素变成更高层的视觉表示。

## 从像素到视觉 Token

现代视觉 Transformer 通常会把图像切成 Patch。

以 ViT 常见设置为例：

```text
输入图像：224 x 224
Patch 大小：16 x 16
Patch 数量：14 x 14 = 196
```

每个 Patch 会被展平、投影成一个向量，类似文本里的 Token Embedding。

```text
图像
  -> Patch 切分
  -> Patch Embedding
  -> 加位置编码
  -> Transformer Encoder
  -> 视觉 Token / 全局特征
```

这就是 ViT 论文标题 “An Image is Worth 16x16 Words” 的核心含义：把图像 Patch 当作类似词元的序列来处理。

## 视觉 Token 数为什么重要？

视觉 Token 数直接影响多模态模型的上下文占用、显存和推理延迟。

| 输入分辨率 | Patch 大小 | Patch 网格 | 视觉 Token 数 |
| ---: | ---: | ---: | ---: |
| 224 x 224 | 16 x 16 | 14 x 14 | 196 |
| 336 x 336 | 16 x 16 | 21 x 21 | 441 |
| 448 x 448 | 16 x 16 | 28 x 28 | 784 |
| 1024 x 1024 | 16 x 16 | 64 x 64 | 4096 |

分辨率越高，模型能看到更多细节，但视觉 Token 数也会快速增加。

这会带来几个后果：

- LLM 上下文被视觉 Token 占用。
- Attention 计算成本上升。
- 多图、多页 PDF、长视频会更贵。
- OCR、图表、GUI 任务需要更高分辨率，成本更难控制。

所以视觉编码器不是“越大越好、分辨率越高越好”。真实系统要在细节保真和推理成本之间取舍。

## CNN、ViT 和 Swin Transformer

视觉编码器有几条典型路线。

| 路线 | 代表 | 核心思路 | 适合理解 |
| --- | --- | --- | --- |
| CNN | ResNet、EfficientNet | 用卷积提取局部纹理和层级特征 | 传统视觉骨干 |
| ViT | Vision Transformer | 把图像切成 Patch，用 Transformer 建模全局关系 | 视觉 Token 化 |
| Swin Transformer | Swin | 用窗口注意力和层级结构兼顾效率与多尺度 | 高分辨率与密集预测 |
| CLIP 系视觉编码器 | CLIP ViT、OpenCLIP、SigLIP | 用图文对齐训练视觉表示 | VLM / MLLM 常用入口 |
| 自监督视觉模型 | DINO、DINOv2 | 不依赖文本监督，学习通用视觉特征 | 分割、定位、密集视觉任务 |

### CNN

CNN 的优势在于局部归纳偏置强，适合图像中的纹理、边缘、局部模式建模。

它曾长期是视觉任务的默认骨干。ResNet 通过残差连接显著改善深层网络训练，是传统视觉编码器的重要基础。

### ViT

ViT 的关键转变是把图像变成 Patch 序列，然后使用 Transformer 处理。

它弱化了卷积先验，更依赖大规模数据预训练。对于 MLLM 来说，ViT 的序列化形式天然更容易和 LLM 的 Token 序列对接。

### Swin Transformer

标准 ViT 对高分辨率图像的全局注意力成本较高。Swin Transformer 使用窗口注意力和层级结构，在效率和多尺度表示之间做平衡。

这类结构对检测、分割、文档、遥感等高分辨率场景很有启发。

## CLIP 系视觉编码器

CLIP 是多模态视觉编码器里绕不开的起点。

它训练两个编码器：

```text
图像编码器：image -> image embedding
文本编码器：text -> text embedding
```

训练目标是让匹配的图文对更接近，不匹配的图文对更远。

```text
相似图文对：靠近
不相似图文对：拉远
```

CLIP 的重要性不只是图文检索，而是它让视觉表示具备了开放词表语义能力。

例如模型不只会判断固定类别，还能把图片和自然语言描述放到同一个语义空间中比较。这为后来的 VLM、图文检索、多模态 RAG 和视觉指令模型打了基础。

## SigLIP、DINOv2、EVA-CLIP

CLIP 之后，视觉编码器继续沿着不同方向演进。

| 模型 | 重点 | 对多模态系统的意义 |
| --- | --- | --- |
| SigLIP | 用 sigmoid loss 改进图文预训练目标 | 提升图文对齐训练的效率与可扩展性 |
| DINOv2 | 大规模自监督视觉特征 | 对密集视觉、定位、分割类任务更友好 |
| EVA-CLIP | 改进 CLIP 训练策略与规模化视觉表征 | 常被用作强视觉编码器候选 |
| OpenCLIP | 开源 CLIP 训练与模型生态 | 方便复现、替换和工程集成 |

不同视觉编码器的偏好不同：

- 图文检索和开放词表识别，通常更依赖图文对齐式视觉编码器。
- OCR、文档、图表、GUI，需要更高分辨率和细粒度视觉能力。
- 分割、定位、区域理解，可能更依赖密集视觉特征。
- 视频任务还要考虑帧级特征和时序建模。

## 全局特征和局部特征

视觉编码器输出不止一种形式。

| 输出形式 | 说明 | 常见用途 |
| --- | --- | --- |
| 全局特征 | 一张图压缩成一个向量 | 图文检索、分类、相似度计算 |
| Patch 特征 | 每个 Patch 一个向量 | VQA、细粒度理解、LLM 接入 |
| 区域特征 | 对象或区域级表示 | Grounding、检测、区域问答 |
| 多尺度特征 | 不同分辨率层级特征 | 检测、分割、文档理解 |

很多早期图文模型偏好全局特征。现代 MLLM 更常使用 Patch 特征，因为 LLM 需要从图像中读取细节。

例如，问“图中右下角的小字是什么？”时，全局特征通常不够用；模型需要细粒度视觉 Token 或更高分辨率输入。

## 视觉编码器如何进入 LLM？

视觉编码器本身输出的特征，通常不能直接喂给 LLM。

原因是：

- 维度可能不同。
- 表示空间不同。
- 视觉 Token 数可能太多。
- LLM 需要的是能和文本上下文交互的表示。

因此中间通常会有一个连接器：

```text
视觉编码器
  -> 视觉特征
  -> Projector / Q-Former / Resampler / Cross-Attention
  -> LLM 可接收的视觉 Token
```

这部分会在 [跨模态对齐与连接器](../04_跨模态对齐与连接器/README.md) 章节详细展开。

本章只需要记住一点：视觉编码器负责“提取视觉信息”，连接器负责“把视觉信息翻译成 LLM 能用的形式”。

## 不同任务对视觉编码器的要求

视觉编码器没有绝对最优，只有任务匹配。

| 任务 | 视觉编码器更需要什么 |
| --- | --- |
| 图文检索 | 强图文对齐、全局语义表示 |
| 图像问答 | 语义理解、局部细节、与 LLM 对接稳定 |
| OCR / 文档理解 | 高分辨率、小字识别、版面结构 |
| 图表理解 | 文字、坐标、颜色、图形结构 |
| GUI Agent | 控件识别、布局理解、可操作区域定位 |
| 视频理解 | 帧级特征、时间采样、动作和事件信息 |
| 医学 / 工业图像 | 细粒度视觉差异和领域适配 |

因此看一个 MLLM 的视觉部分时，不要只问“用了哪个视觉编码器”，还要问它要解决什么任务。

## 视频中的视觉编码器

视频可以看成连续图像帧，但不能只当作一堆图片。

视频多了一个关键维度：时间。

```text
视频
  -> 抽帧
  -> 每帧视觉编码
  -> 时序聚合
  -> 视频表示 / LLM 输入
```

视频任务有几个额外问题：

- 抽多少帧？
- 关键动作是否被采到？
- 帧之间的顺序是否被保留？
- 长视频如何控制 Token 数？
- 模型能否理解事件变化，而不只是单帧内容？

这部分会在 [视频多模态](../07_视频多模态/README.md) 章节继续展开。

## 文档、截图和 GUI 的特殊性

文档和 GUI 截图对视觉编码器很苛刻。

它们往往包含：

- 小字号文字
- 表格线和版面结构
- 图表、坐标轴和图例
- 按钮、菜单、输入框
- 多栏布局和阅读顺序

普通自然图像编码器不一定擅长这些任务。

因此文档理解、GUI Agent 常常需要：

- 更高输入分辨率
- OCR 或端到端文字识别能力
- 布局信息建模
- 局部区域放大
- 多尺度视觉特征

这也是为什么“能看图”不等于“能读文档”。

## 常见误区

### 1. 视觉编码器越大越好

更大的编码器通常能力更强，但也更慢、更占显存。实际系统要看任务、成本和延迟。

### 2. 分辨率越高越好

高分辨率能保留细节，但视觉 Token 数会快速上升。多图、多页、长视频场景尤其要控制成本。

### 3. CLIP 强就代表所有视觉任务都强

CLIP 擅长图文对齐和开放词表语义，但不一定天然擅长 OCR、精确定位、细粒度计数和文档结构理解。

### 4. 视觉编码器决定全部多模态能力

视觉编码器很重要，但连接器、LLM、训练数据、指令微调和评测同样关键。

### 5. 图片能进模型，就代表模型真的看懂了

模型可能识别到大致场景，但忽略细节、位置关系或文字内容。是否“看懂”，要靠任务和评测验证。

## 前瞻性判断

视觉编码器未来仍会长期演进，但有几条方向比较确定。

### 1. 更高分辨率和更低 Token 成本会并行发展

OCR、文档、GUI、医学和工业场景需要高分辨率；但高分辨率会带来大量视觉 Token。

所以后续会继续发展：

- 动态分辨率
- Token 压缩
- 局部区域选择
- 多尺度表示
- 重要区域重采样

### 2. 通用视觉特征和任务专用视觉能力会共存

CLIP、DINOv2 这类通用表征很重要，但真实业务常常需要领域适配。

例如医学影像、遥感图像、工业缺陷检测、GUI 自动化，都可能需要专门数据和专门评测。

### 3. 视觉编码器会从单图走向视频、文档和界面

下一阶段的视觉输入不只是照片，还会更多来自：

- PDF 页面
- 网页截图
- 移动端界面
- 长视频
- 多摄像头画面
- 机器人视角

视觉编码器需要更强的结构理解和时序理解。

### 4. Grounding 会变得更重要

模型不只要回答“是什么”，还要能指出“在哪里”“依据是什么”。

区域定位、引用、框选、时间戳、文档页码，都会成为多模态系统可靠性的一部分。

## 本章小结

视觉编码器的核心可以概括为：

```text
像素输入
  -> Patch / 区域 / 帧
  -> 视觉特征
  -> 视觉 Token
  -> 连接器
  -> LLM 或多模态生成模块
```

学习视觉编码器，不要只记模型名。更重要的是理解它在多模态系统里负责什么、输出什么、成本如何、适合什么任务，以及它不能解决什么问题。

## 推荐阅读与引用来源

- [Deep Residual Learning for Image Recognition](https://arxiv.org/abs/1512.03385)：ResNet，传统 CNN 视觉骨干的重要基础。
- [An Image is Worth 16x16 Words: Transformers for Image Recognition at Scale](https://arxiv.org/abs/2010.11929)：ViT，把图像切成 Patch 序列并使用 Transformer 处理。
- [Swin Transformer: Hierarchical Vision Transformer using Shifted Windows](https://arxiv.org/abs/2103.14030)：Swin Transformer，用窗口注意力和层级结构提升高分辨率视觉建模效率。
- [Learning Transferable Visual Models From Natural Language Supervision](https://arxiv.org/abs/2103.00020)：CLIP，图文对比学习和开放词表视觉能力的重要基础。
- [Sigmoid Loss for Language Image Pre-Training](https://arxiv.org/abs/2303.15343)：SigLIP，用 sigmoid loss 改进图文预训练目标。
- [DINOv2: Learning Robust Visual Features without Supervision](https://arxiv.org/abs/2304.07193)：DINOv2，大规模自监督视觉表征。
- [EVA-CLIP: Improved Training Techniques for CLIP at Scale](https://arxiv.org/abs/2303.15389)：EVA-CLIP，改进 CLIP 训练策略和规模化视觉表征。
