# 13_实战项目

> 实战的目的不是把模型调用成功，而是完成一个可复现、可评测、可解释的多模态闭环。

**By：猫先生 of 「魔方AI空间」**

## 本章定位

本章把前面的概念落到项目中。

推荐按以下顺序推进：

```text
单模态能力验证
  -> 跨模态检索
  -> 多模态问答
  -> 文档 / 视频 / 音频系统
  -> 评测与错误分析
  -> Agent 闭环
```

项目不绑定某个模型版本。视觉编码器、VLM、LLM、向量库和推理服务都应通过清晰接口替换。

## 什么算完成一个项目？

一个项目至少包含：

| 项目 | 要求 |
| --- | --- |
| 目标 | 明确验证哪项能力 |
| 输入输出 | 定义数据格式和边界 |
| 最小闭环 | 从输入到结果完整可运行 |
| 基线 | 有简单方法作为对照 |
| 评测 | 有指标和固定样例集 |
| 错误分析 | 保留失败样例并分类 |
| 可复现 | 记录环境、模型、参数和数据版本 |
| 局限 | 说明哪些结论不能外推 |

只有截图，没有指标和失败分析，不算完整实验。

## 项目总览

| 编号 | 项目 | 核心能力 | 难度 |
| ---: | --- | --- | --- |
| P0 | 最小图像问答 | VLM 推理与证据核对 | 入门 |
| P1 | 图文语义检索 | 跨模态表示与召回 | 入门 |
| P2 | 带引用的文档问答 | OCR、版面、RAG、证据 | 进阶 |
| P3 | 视频与音频时间问答 | 抽帧、转写、时间定位 | 进阶 |

| P6 | GUI Agent 沙盒 | Grounding、行动、验证 | 高阶 |
| P7 | 多模态任务助手 | 多模态、RAG、工具闭环 | 综合 |

## P0：最小图像问答

### 目标

跑通最小 VLM 推理，并验证模型是否真的使用图像。

### 输入输出

```text
输入：图像 + 问题
输出：答案 + 简短证据描述
```

### 最小功能

- 支持单张图像。
- 支持固定 Prompt。
- 保存原始请求和输出。
- 记录延迟、Token 和模型版本。

### 必做实验

1. 正确图像 + 正确问题。
2. 去掉图像。
3. 替换为错误图像。
4. 只修改一个视觉属性。
5. 输入模糊或无法回答的图像。

### 验收标准

- 更换关键视觉事实时，答案同步变化。
- 无图或错误图像时，模型不能保持相同高置信回答。
- 看不清时能够表达不确定。

### 对应章节

[视觉语言模型 VLM](../05_视觉语言模型VLM/README.md) 与 [多模态评测](../10_多模态评测/README.md)。

## P1：图文语义检索

### 目标

理解图像和文本如何进入共享向量空间。

### 系统流程

```text
图片库
  -> 图像编码
  -> 向量索引

文本查询
  -> 文本编码
  -> 相似度检索
  -> Top-k 图片
```

### 最小功能

- 建立小型图片库。
- 支持文本搜图和以图搜图。
- 返回 Top-k 结果与相似度。
- 可视化正确与错误召回。

### 评测

- Recall@1
- Recall@5
- MRR
- 平均检索延迟

### 必做对照

- 简单关键词检索。
- 不同 Caption 详细程度。
- 相似类别中的细粒度属性。
- 中英文查询或同义改写。

### 验收标准

不仅能区分类别，还能区分颜色、动作、数量和关系。

### 对应章节

[视觉编码器](../02_视觉编码器/README.md) 与 [跨模态对齐与连接器](../04_跨模态对齐与连接器/README.md)。

## P2：带引用的文档问答

### 目标

构建能够回答 PDF、报告或表格问题，并返回证据位置的系统。

### 系统流程

```text
文档
  -> 页面渲染与解析
  -> OCR / 版面识别
  -> 文本、页面和区域索引
  -> 检索相关证据
  -> VLM / LLM 回答
  -> 页码与区域引用
```

### 最小功能

- 支持多页文档。
- 同时保留文本和页面图像。
- 答案附带页码。
- 无证据时明确拒答。

### 评测集

至少覆盖：

- 正文事实
- 表格字段
- 图表趋势
- 跨页问题
- 无答案问题
- 易混淆数字

### 核心指标

| 指标 | 检查内容 |
| --- | --- |
| Answer Accuracy | 答案是否正确 |
| Retrieval Recall | 正确证据是否被召回 |
| Citation Accuracy | 页码和区域是否正确 |
| Abstention Accuracy | 无答案时是否拒答 |
| Latency | 端到端耗时 |

### 验收标准

回答、引用和原文三者能够互相核对。

### 对应章节

[多模态数据与训练](../09_多模态数据与训练/README.md) 与 [多模态 Agent 与应用](../11_多模态Agent与应用/README.md)。

## P3：视频与音频时间问答

### 目标

让系统回答视频内容问题，并返回相关时间段。

### 系统流程

```text
视频
  -> 镜头切分 / 抽帧
  -> 音频转写
  -> 帧、字幕和时间戳索引
  -> 问题相关片段检索
  -> 多模态回答
  -> 时间引用
```

### 最小功能

- 支持短视频。
- 保存帧与字幕时间戳。
- 返回答案和证据片段。
- 可播放或跳转到证据位置。

### 必做实验

- 均匀抽帧和镜头抽帧对比。
- 仅画面、仅字幕、音画联合对比。
- 短暂事件和长程事件对比。
- 加入无关片段后的鲁棒性。

### 核心指标

- 问答准确率
- 时间定位命中率
- 关键事件召回率
- 不同抽帧预算下的延迟

### 验收标准

系统不仅能答，还能指出答案发生在什么时候。

### 对应章节

[视频多模态](../07_视频多模态/README.md) 与 [音频语音多模态](../08_音频语音多模态/README.md)。

## P4：多模态评测工具箱

### 目标

建立一个可重复运行的模型对比和回归评测工具。

### 数据结构

每条样例至少包含：

```json
{
  "id": "sample-001",
  "media": ["path/to/image.png"],
  "question": "图中有几个人？",
  "reference": "2",
  "capability": "counting",
  "answerable": true
}
```

### 最小功能

- 统一模型调用接口。
- 固定 Prompt 和推理参数。
- 保存原始输出。
- 自动评分与人工复核并存。
- 按能力输出分项结果。
- 对版本变化做回归比较。

### 测试维度

- 感知
- OCR
- 空间关系
- 多图比较
- 幻觉
- 无答案
- Prompt 改写
- 图像压缩和裁剪
- 延迟与成本

### 验收标准

同一评测集可稳定比较两个模型或同一模型的两个版本，并能定位能力回退。

### 对应章节

[多模态评测](../10_多模态评测/README.md)。

## P5：可控内容生成与编辑

### 目标

构建文本与图像条件共同控制的生成或编辑流程。

### 推荐任务

从以下任务任选一个：

- 局部重绘
- 背景替换
- 草图控制生成
- 姿态控制生成
- 参考图风格生成
- 首帧生成短视频

### 最小闭环

```text
原始内容 + 编辑指令 + 控制条件
  -> 生成 / 编辑
  -> 自动检查
  -> 人工选择
  -> 保存参数和结果
```

### 评测重点

- 指令是否生效。
- 未编辑区域是否保持。
- 主体身份是否一致。
- 多次生成是否稳定。
- 速度和显存是否可接受。

### 验收标准

不能只展示最好的一张。应固定一组输入，报告成功率和典型失败。

### 对应章节

[多模态生成模型](../06_多模态生成模型/README.md)。

## P6：GUI Agent 沙盒

### 目标

让 Agent 在受控界面中完成多步任务。

### 环境要求

- 使用测试账号和测试数据。
- 禁止真实付款、发送和删除。
- 所有动作可记录、可重放。
- 高风险动作必须确认。

### Agent 闭环

```text
截图 / UI 结构
  -> 当前状态
  -> 控件 Grounding
  -> 动作规划
  -> 点击 / 输入 / 滚动
  -> 新观察
  -> 结果验证
```

### 最小任务

- 打开目标页面。
- 找到指定控件。
- 填写两个字段。
- 提交到测试环境。
- 验证成功提示和数据状态。

### 核心指标

| 指标 | 含义 |
| --- | --- |
| Task Success | 完整任务是否成功 |
| Grounding Accuracy | 控件是否定位正确 |
| Step Count | 是否存在多余步骤 |
| Recovery Rate | 错误后能否恢复 |
| Unsafe Action Rate | 是否执行越权动作 |

### 验收标准

任务成功必须由环境状态验证，不能由模型自行宣布。

### 对应章节

[多模态 Agent 与应用](../11_多模态Agent与应用/README.md)。

## P7：多模态任务助手

### 目标

把前面的能力整合为一个可用系统。

### 示例场景

用户上传一份带图表的报告和一段会议视频，系统完成：

1. 提取文档结构。
2. 生成带页码引用的摘要。
3. 定位视频中的相关讨论。
4. 对比文档和会议结论。
5. 生成待办事项。
6. 经用户确认后写入任务系统。

### 系统结构

```text
多模态输入
  -> 解析与索引
  -> 检索相关证据
  -> 多模态理解
  -> 任务规划
  -> 工具调用
  -> 结果验证
  -> 带引用输出
```

### 质量门槛

- 所有关键结论有证据引用。
- 工具调用前进行参数校验。
- 写入和发送前需要确认。
- 无答案时不编造。
- 失败时保留可读日志。

### 验收标准

用完整业务任务成功率验收，不以单轮回答质量代替。

## 推荐项目目录

```text
project-name/
├── README.md
├── configs/
├── data/
│   ├── raw/
│   ├── processed/
│   └── eval/
├── src/
│   ├── adapters/
│   ├── pipeline/
│   ├── evaluation/
│   └── utils/
├── outputs/
├── tests/
└── experiment-log.md
```

目录职责：

| 目录 | 内容 |
| --- | --- |
| `configs/` | 模型、Prompt、采样与运行参数 |
| `data/raw/` | 原始输入，只读保存 |
| `data/processed/` | 解析、切分和索引产物 |
| `data/eval/` | 固定评测样例 |
| `src/adapters/` | 可替换的模型和工具接口 |
| `src/pipeline/` | 业务流程 |
| `src/evaluation/` | 指标和回归评测 |
| `outputs/` | 原始输出、日志和报告 |
| `tests/` | 单元、集成和最小回归测试 |

## 统一接口

尽量把模型能力封装为稳定接口：

```python
class MultimodalModel:
    def generate(self, prompt, media, **options):
        ...


class Retriever:
    def index(self, records):
        ...

    def search(self, query, top_k=5):
        ...


class Tool:
    def validate(self, arguments):
        ...

    def execute(self, arguments):
        ...
```

接口稳定后，底层模型可以替换，项目评测和业务逻辑仍然保留。

## 统一实验记录

每次实验使用 [实验记录模板](../templates/experiment-log-template.md)，至少记录：

- 日期和负责人
- 代码提交或版本
- 模型和数据版本
- Prompt
- 推理参数
- 硬件与环境
- 指标结果
- 成功和失败样例
- 结论与下一步

不要覆盖旧结果。实验记录应能解释一次改动为什么变好或变差。

## 最小评测集

每个项目从少量固定样例开始。

建议结构：

| 样例类型 | 作用 |
| --- | --- |
| 简单正例 | 验证基本链路 |
| 困难正例 | 验证能力上限 |
| 反事实样例 | 检查是否依赖真实证据 |
| 无答案样例 | 检查不确定性 |
| 噪声样例 | 检查鲁棒性 |
| 安全样例 | 检查权限和拒答 |
| 历史失败样例 | 防止版本回退 |

评测集不需要一开始很大，但必须固定、可复现并持续增长。

## 实战中的五条原则

### 1. 先做基线

先用最简单的方法跑通，再增加复杂模块。

### 2. 先定义成功

没有验收标准，就无法判断模型或系统是否改进。

### 3. 保留原始输出

只保存最终分数，会失去错误分析依据。

### 4. 模型与业务解耦

通过 Adapter 隔离模型差异，避免项目随版本失效。

### 5. 从失败样例迭代

真实进步来自修复稳定可复现的失败，不来自挑选成功截图。

## 推荐推进顺序

```text
第 1 阶段：P0 + P1
  建立推理、向量和基础评测能力

第 2 阶段：P2 + P3
  处理文档、视频、音频和证据引用

第 3 阶段：P4
  建立统一评测与回归工具

第 4 阶段：P5 或 P6
  进入生成工作流或 Agent 行动闭环

第 5 阶段：P7
  完成综合系统
```

不要一开始就做 P7。先把可观察、可评测的基础模块做稳。

## 本章小结

实战主线可以压缩为：

```text
明确目标
  -> 建立最小闭环
  -> 设置基线和评测集
  -> 保存结果与失败样例
  -> 通过稳定接口替换模型
  -> 做回归验证
  -> 再扩展能力
```

真正能跨越模型周期的，不是某次 API 调用代码，而是清晰接口、固定评测、实验记录和错误分析。
