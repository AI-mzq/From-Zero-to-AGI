# 03_文本编码器与 LLM 底座

> 在多模态大模型里，视觉编码器负责“看”，连接器负责“翻译”，LLM 底座负责理解指令、组织上下文、进行推理，并生成最终回答。

**By：猫先生 of 「魔方AI空间」**

## 本章导读

前一章讲了 [视觉编码器](../02_视觉编码器/README.md)：图像、截图、文档页或视频帧要先被编码成视觉特征。

但视觉特征本身还不能完成对话、推理、总结、工具调用。真正把这些视觉信息组织成自然语言答案的，通常是 LLM 底座。

一个典型流程是：

```text
图像 / 视频 / 文档
  -> 视觉编码器
  -> 连接器
  -> LLM 底座
  -> 回答 / 推理 / 工具调用
```

这一章重点回答：

- 多模态模型里的“文本编码器”和“LLM 底座”分别指什么？
- Decoder-only LLM 为什么成为 MLLM 的常见中枢？
- 文本 Token、视觉 Token 和上下文窗口是什么关系？
- Base Model、Chat Model、Reasoning Model 对多模态能力有什么影响？
- 为什么多模态模型需要视觉指令微调？
- LLM 底座强，是否就意味着多模态能力一定强？

## 先说清：文本编码器和 LLM 底座不是一回事

早期视觉语言模型里，经常会有一个文本编码器。

例如 CLIP 会分别训练：

```text
图像编码器：image -> image embedding
文本编码器：text -> text embedding
```

文本编码器的目标，是把文本变成一个可比较的向量，用来和图像向量做相似度计算。

但现代 MLLM 更常见的是另一种结构：

```text
视觉特征
  -> 连接器
  -> LLM 底座
  -> 自回归生成回答
```

这里的 LLM 底座不只是编码文本，它还负责生成、推理、对话、指令遵循和工具调用。

简单区分如下：

| 组件 | 常见位置 | 主要作用 |
| --- | --- | --- |
| 文本编码器 | CLIP、ALIGN 等图文对齐模型 | 把文本编码成向量，便于图文匹配 |
| LLM 底座 | LLaVA、Qwen-VL、InternVL 等 MLLM | 处理多模态上下文并生成回答 |
| 文本解码器 | Caption、生成模型、Decoder-only LLM | 按 Token 生成文本 |

本章重点讨论现代多模态大模型里的 LLM 底座。

## LLM 在多模态系统里负责什么？

LLM 不是负责“看”的模块，它更像多模态系统的语言和推理中枢。

| 能力 | LLM 底座的作用 |
| --- | --- |
| 指令理解 | 理解用户想让模型做什么 |
| 上下文组织 | 把文本、视觉 Token、历史对话放进同一上下文 |
| 语言生成 | 生成自然语言回答、摘要、解释、代码或结构化输出 |
| 推理 | 基于输入信息进行比较、归纳、计算和多步判断 |
| 对话 | 维护多轮上下文和指代关系 |
| 工具调用 | 决定何时调用 OCR、检索、代码、浏览器等工具 |
| 安全边界 | 在高风险、不确定或越权场景下控制输出 |

换句话说，视觉编码器把“看见的东西”变成表示，LLM 决定如何把这些表示用于任务。

## 为什么常用 Decoder-only LLM？

现代 MLLM 很多都选择 Decoder-only LLM 作为底座，例如 LLaMA、Qwen、Mistral、Vicuna、DeepSeek 等。

原因很直接：

- 它适合自回归生成。
- 它已经具备较强语言理解和指令遵循能力。
- 它天然支持多轮对话和长上下文。
- 它可以接入工具调用、RAG、Agent 等应用模式。
- 开源生态成熟，便于微调和部署。

一个简化的 Decoder-only 多模态输入可以理解为：

```text
<system>
你是一个多模态助手。

<user>
请看这张图，解释图表趋势。

<image_tokens>
v1, v2, v3, ..., vn

<assistant>
根据图表可以看到...
```

这里的 `<image_tokens>` 不是普通文字，而是由视觉编码器和连接器生成的一组向量表示。LLM 会把它们作为上下文的一部分参与后续生成。

## 文本 Token 与视觉 Token

LLM 的上下文窗口不是只装文本。多模态模型里，视觉信息也会占用上下文预算。

| Token 类型 | 来源 | 说明 |
| --- | --- | --- |
| 文本 Token | Tokenizer | 用户问题、系统提示词、历史对话 |
| 视觉 Token | 视觉编码器 + 连接器 | 图像 Patch、区域、帧或压缩后的视觉表示 |
| 特殊 Token | 模型模板 | `<image>`、`<video>`、`<bbox>`、分隔符等 |
| 工具 Token | 工具调用协议 | 函数参数、检索结果、执行反馈 |

视觉 Token 数越多，模型能接收的文本上下文就越少，推理成本也越高。

例如：

```text
上下文窗口：32K Token
文本提示：2K Token
一张高分辨率图片：1K 到数千视觉 Token
多页 PDF / 长视频：可能迅速吃满上下文
```

所以 MLLM 不只是要“看得清”，还要“看得省”。

## 上下文窗口和视觉预算

多模态系统经常会遇到一个朴素但重要的问题：

> 视觉信息到底占多少上下文？

如果视觉 Token 太少，模型可能看不清细节；如果视觉 Token 太多，成本和延迟会上升，还会挤压文本上下文。

| 场景 | 视觉预算挑战 |
| --- | --- |
| 单图问答 | 需要平衡整体语义和局部细节 |
| OCR / 文档 | 小字和版面需要高分辨率，Token 数容易上升 |
| 多图对比 | 每张图都要占视觉 Token，容易挤压文本 |
| 长视频 | 帧数、分辨率和时序信息同时占预算 |
| GUI Agent | 截图细节、控件位置和历史操作都要保留 |

这也是为什么后续章节会讲视觉 Token 压缩、连接器、动态分辨率和长上下文策略。

## Base Model、Chat Model、Reasoning Model

同样是 LLM 底座，不同训练阶段会带来不同能力。

| 类型 | 特点 | 对多模态系统的影响 |
| --- | --- | --- |
| Base Model | 主要学习语言分布，未充分对齐指令 | 可作为底座，但直接对话能力通常不足 |
| Chat / Instruct Model | 经过指令微调和对齐 | 更适合多模态问答、助手、文档理解 |
| Reasoning Model | 强化数学、代码、复杂推理 | 对图表推理、视觉推理、多步任务更有帮助 |

很多多模态模型不是从零训练 LLM，而是选一个已有 LLM，再接入视觉编码器和连接器，最后做多模态训练。

这意味着：

- LLM 的语言能力会影响回答质量。
- LLM 的指令遵循能力会影响任务完成度。
- LLM 的推理能力会影响视觉推理和图表理解。
- LLM 的安全对齐会影响高风险场景输出。

但也要注意：强 LLM 不会自动变成强 MLLM。视觉编码器、连接器和多模态数据仍然决定模型能不能真正利用视觉信息。

## 为什么需要视觉指令微调？

只把视觉 Token 接到 LLM 前面，模型不一定知道该怎么用。

视觉指令微调要做的事情，是教模型把视觉输入和人类指令联系起来。

例如：

```text
输入：图片 + “这张图里有什么异常？”
输出：指出异常区域，并用自然语言解释
```

视觉指令数据通常包含：

- 图像描述
- 图像问答
- OCR 问答
- 多轮图文对话
- 图表理解
- 文档理解
- 视觉推理
- 安全拒答和不确定性表达

没有这一步，模型可能能接收图像特征，但不一定会按用户指令使用它们。

LLaVA 的视觉指令微调路线，就是这一方向的代表工作之一。

## LLM 底座如何影响多模态任务？

不同任务对 LLM 底座的依赖不一样。

| 任务 | 更依赖 LLM 的什么能力 |
| --- | --- |
| 图像描述 | 语言组织、概括、细节选择 |
| VQA | 指令理解、视觉信息整合、短答案生成 |
| OCR 问答 | 文本理解、引用、格式保持 |
| 图表理解 | 数值推理、趋势解释、结构化表达 |
| 多图对比 | 上下文管理、比较和归纳 |
| 文档问答 | 长上下文、引用、事实一致性 |
| GUI Agent | 指令分解、状态跟踪、工具调用 |
| 视觉推理 | 多步推理和不确定性控制 |

所以评估一个 MLLM 时，不只看视觉 Benchmark，也要看底座 LLM 本身的语言、推理和对齐能力。

## LLM 不是万能中枢

虽然 LLM 很关键，但它也有边界。

### 1. 看不清的内容，LLM 很难补救

如果视觉编码器没有捕捉到小字、区域、动作或细节，LLM 可能会猜测，甚至产生幻觉。

### 2. 连接器对齐不好，LLM 会误解视觉信息

视觉特征需要映射到 LLM 能理解的空间。连接器训练不足时，LLM 可能无法稳定利用视觉 Token。

### 3. 训练数据会塑造模型习惯

如果训练数据偏向简短 Caption，模型可能擅长描述；如果数据包含高质量视觉指令，模型更可能完成复杂问答。

### 4. 推理能力不能替代证据

LLM 可以推理，但不能把图中不存在的信息推出来。多模态任务必须回到输入证据。

## Prompt 在多模态里的作用

多模态 Prompt 不只是文字提示词，还包括输入顺序、图像位置、历史对话和输出格式。

一个简单模板可能是：

```text
系统角色：
你是一个谨慎的多模态助手。

输入：
<image>
用户问题：请指出图中表格的主要趋势。

要求：
1. 只基于图片内容回答。
2. 如果图片信息不足，请说明无法判断。
3. 用三点概括。
```

这个 Prompt 会影响模型如何使用视觉信息：

- 是否关注图像证据
- 是否避免编造
- 是否按格式输出
- 是否在不确定时说明边界

多模态 Prompt 工程会在后续应用章节继续展开。

## 多模态中的特殊 Token

多模态模型常使用特殊 Token 标记不同输入区域。

| 特殊 Token | 作用 |
| --- | --- |
| `<image>` | 标记图像输入位置 |
| `<video>` | 标记视频输入位置 |
| `<audio>` | 标记音频输入位置 |
| `<bbox>` | 标记边界框或区域信息 |
| `<ref>` | 标记引用对象或区域 |
| `<tool_call>` | 标记工具调用请求 |

不同模型的具体格式不同，但思想类似：让 LLM 知道上下文里哪些部分来自非文本模态。

工程上要特别注意，模板格式、特殊 Token 和训练数据必须匹配。否则模型可能无法正确识别输入边界。

## 常见误区

### 1. LLM 强，多模态就一定强

不一定。LLM 强只能说明语言和推理底座好，多模态还需要视觉编码器、连接器和多模态数据。

### 2. 视觉 Token 可以无限塞

视觉 Token 会占上下文、显存和推理时间。多图、长视频、PDF 场景尤其要控制预算。

### 3. 多模态只是把图像特征拼到 Prompt 里

拼接只是形式。关键在于视觉特征是否被正确对齐，LLM 是否学会按指令使用它们。

### 4. Chat Model 一定适合所有多模态任务

Chat Model 擅长对话，但文档、图表、GUI、医学等任务可能需要专门数据和评测。

### 5. 推理模型可以解决所有视觉问题

推理模型能改善多步判断，但如果视觉证据缺失或错误，推理只会把错误包装得更合理。

## 前瞻性判断

### 1. LLM 底座会继续向长上下文和强推理演进

多图、多页文档、长视频和 Agent 任务都需要更长上下文。视觉 Token 越多，长上下文能力越重要。

### 2. 多模态模型会更重视证据引用

未来模型不只要回答，还要说明答案来自哪张图、哪一页、哪一块区域、哪一段视频。

这会让 LLM 底座和 Grounding、RAG、引用系统结合得更紧。

### 3. 视觉输入会和工具调用更深结合

LLM 不会只看图回答，还会根据图像或界面决定是否调用 OCR、检索、计算器、代码执行、浏览器或外部工具。

### 4. 底座模型的安全边界会更重要

多模态系统会处理截图、证件、医疗影像、合同、监控视频等敏感内容。LLM 底座必须能处理隐私、安全、拒答和不确定性。

## 本章小结

LLM 底座在多模态系统中的作用可以概括为：

```text
视觉 / 音频 / 视频特征
  -> 连接器映射
  -> 与文本上下文融合
  -> LLM 理解指令和组织推理
  -> 生成回答或调用工具
```

学习这一章时，重点不是把 LLM 说成万能大脑，而是理解它在多模态系统中的职责边界：它负责语言、推理、对话和工具调用，但能否真正利用视觉证据，还要看视觉编码器、连接器、训练数据和评测体系。
