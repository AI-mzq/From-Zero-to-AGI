# 10_多模态评测

> 多模态评测不是给模型做一套选择题，而是验证它是否真的使用了图像、视频或音频证据，并在真实任务中稳定完成目标。

**By：猫先生 of 「魔方AI空间」**

## 本章导读

上一章讨论了 [多模态数据与训练](../09_多模态数据与训练/README.md)。训练决定模型学什么，评测决定我们是否真的知道它学会了什么。

多模态评测比纯文本评测多一层困难：

```text
答案正确
  不一定代表模型看懂了输入
```

模型可能依赖语言常识、数据记忆或题目线索猜对答案。严谨评测必须区分：

- 看见了什么。
- 是否理解模态内容。
- 是否根据证据推理。
- 回答是否忠实、稳定和安全。

本章抓住六条主线：

1. 测什么能力？
2. 样本能否隔离目标能力？
3. 指标是否真的衡量目标？
4. 模型是否使用了模态证据？
5. 评测是否被污染或 Judge 偏差影响？
6. 离线高分能否转化为真实可用性？

## 一句话理解多模态评测

多模态评测的核心是：

> 用可解释、可复现的任务，判断模型是否基于正确的模态证据，在明确约束下完成目标。

一套评测至少要回答四件事：

| 问题 | 含义 |
| --- | --- |
| 测什么 | 感知、理解、推理、生成、交互还是安全 |
| 怎么测 | 选择题、开放问答、定位、生成或完整任务 |
| 如何判分 | 规则、模型 Judge、人工还是业务结果 |
| 分数是否可信 | 是否污染、泄漏、提示敏感或偏离真实场景 |

## 先定义能力，再选择 Benchmark

“多模态能力”太宽，必须拆解。

| 能力层 | 核心问题 | 典型任务 |
| --- | --- | --- |
| 感知 | 是否看见或听见关键信息 | 物体、文字、声音、动作识别 |
| 对齐 | 不同模态是否正确对应 | 图文匹配、音画同步、区域指代 |
| 结构 | 是否理解位置、布局和时间 | OCR、文档、图表、时序定位 |
| 推理 | 是否能基于证据得出结论 | 数学、因果、比较、多步问答 |
| 生成 | 输出是否忠实、清楚、可控 | Caption、编辑、图像和视频生成 |
| 交互 | 是否能完成多轮真实任务 | GUI Agent、工具调用、语音对话 |
| 安全 | 是否识别风险并遵守边界 | 隐私、越权、欺骗和有害内容 |

一个总分会掩盖差异。模型可能 OCR 很强，但空间关系很弱；也可能语言表达很好，却经常忽略图像。

## 多模态评测为什么难？

### 1. 输入信息复杂

图片可能低清、遮挡或包含小字；视频很长；音频可能有噪声和多人重叠。

### 2. 正确答案不唯一

开放描述、摘要和生成任务往往存在多个合理答案。

### 3. 语言能力会掩盖感知缺陷

模型可以根据常识生成合理答案，却没有读取输入证据。

### 4. 评分器也会犯错

字符串匹配过于严格，LLM Judge 又可能偏好更长、更流畅或与自己相似的回答。

### 5. Benchmark 容易污染

公开题目、图片和答案可能进入训练数据，使分数高估泛化能力。

## 评测对象：不只是一道题

评测单位会决定你看到哪类能力。

| 评测单位 | 适合检查 |
| --- | --- |
| 单样本 | 基础感知和知识 |
| 多轮对话 | 指代、记忆、纠错和一致性 |
| 长上下文 | 多页文档、长视频和证据检索 |
| 完整任务 | 工具调用、GUI、业务工作流 |
| 长期会话 | 状态保持、用户偏好和错误累积 |

真实系统通常需要从单题评测逐步走向任务级评测。

## 封闭式与开放式评测

### 封闭式评测

选择题、判断题和短答案容易自动判分。

优点：

- 便于批量比较。
- 复现成本低。
- 指标清晰。

局限：

- 可能被选项线索利用。
- 猜测也能得分。
- 难以观察解释质量。

### 开放式评测

模型自由生成答案、描述或分析。

优点：

- 更接近真实使用。
- 能检查表达、推理和证据引用。

局限：

- 判分困难。
- 对 Prompt 和 Judge 更敏感。
- 正确答案可能不唯一。

成熟评测通常同时使用两者。

## 常见任务与指标

| 任务 | 常用指标 | 主要盲点 |
| --- | --- | --- |
| 分类 / 选择题 | Accuracy、F1 | 猜测、选项偏差 |
| VQA | Exact Match、软匹配 | 同义表达和答案粒度 |
| OCR | 字错率、词错率 | 版面和阅读顺序 |
| 检索 | Recall、MRR | 只反映排序，不反映生成 |
| 定位 | IoU、命中率 | 标注边界可能主观 |
| Caption | BLEU、CIDEr、SPICE | 参考答案有限，忠实性不足 |
| 开放问答 | Judge、人工评分 | Judge 偏差和评分不稳定 |
| 生成内容 | 自动指标 + 人类偏好 | 审美和任务价值难统一 |
| Agent | 成功率、步骤数、耗时 | 环境波动和工具错误 |

指标必须匹配任务。不能用语言流畅度代替视觉忠实性，也不能用单帧清晰度代替视频一致性。

## 底层关键：构念效度

构念效度回答一个根本问题：

> 这套测试真的测到了它声称要测的能力吗？

例如，一道题声称评测视觉计数：

```text
图中有几只斑马？
```

但如果数据中绝大多数答案都是“2”，模型可能学会答案先验，而不是计数。

检查构念效度可以问：

- 去掉图像后，模型还能答对多少？
- 打乱图片和问题后，分数下降多少？
- 更换选项顺序后，结果是否稳定？
- 只改一个视觉属性，答案是否随之改变？
- 题目是否同时依赖多个未控制变量？

评测首先是实验设计，其次才是排行榜。

## 模型真的使用了模态证据吗？

可以用消融和反事实检查。

### 输入消融

分别测试：

```text
完整输入
仅文本
仅图像 / 视频 / 音频
错误模态输入
```

如果去掉图像后分数几乎不变，任务可能主要在测语言先验。

### 反事实替换

只改变一个关键视觉事实：

```text
红色杯子 -> 蓝色杯子
两个人 -> 三个人
左侧 -> 右侧
```

模型答案应随证据变化，而不是保持原答案。

### 证据定位

要求模型同时给出：

- 图像区域
- 文档页码
- 表格单元格
- 视频时间戳
- 音频片段

能回答还不够，最好能指出依据在哪里。

## 感知、推理和知识要分开

一道多模态题可能同时依赖三类能力：

```text
感知：读出图表中的数值
推理：计算两者差值
知识：理解指标含义
```

只记录“最终答错”无法定位问题。

更好的评测会保留中间结果：

1. 模型看到了什么？
2. 提取了哪些结构化事实？
3. 如何计算或推理？
4. 最终答案是什么？

这有助于区分感知错误、推理错误和知识错误。

## 多模态幻觉

多模态幻觉是指输出内容没有得到输入模态支持。

常见类型：

| 类型 | 例子 |
| --- | --- |
| 对象幻觉 | 声称图中存在不存在的物体 |
| 属性幻觉 | 颜色、数量、状态错误 |
| 关系幻觉 | 左右、前后、包含关系错误 |
| OCR 幻觉 | 编造图片中的文字 |
| 时间幻觉 | 编造视频中未发生的事件 |
| 声音幻觉 | 声称听到不存在的声音 |
| 因果幻觉 | 把共现错误解释为因果 |

评测幻觉不能只问“答案流畅吗”，而要逐项核对输出声明是否有证据。

POPE、HallusionBench 等工作分别从对象存在性、视觉错觉和语言先验等角度检查这一问题。

## 无答案与不确定性

现实输入经常无法回答：

- 图片太模糊。
- 关键区域被遮挡。
- 视频没拍到事件。
- 音频噪声过大。
- 文档缺少相关页。
- 问题前提本身错误。

评测集必须包含这类样本。

可靠模型应能区分：

```text
我知道
我不确定
输入中没有证据
问题无法回答
```

只奖励“给出答案”，会训练和评测出过度自信的模型。

## 鲁棒性评测

同一能力应在轻微扰动下保持稳定。

可测试：

- 改写问题。
- 更换选项顺序。
- 图片压缩、裁剪、旋转或降采样。
- 视频改变抽帧率。
- 音频加入噪声或改变音量。
- 调整图像在多图输入中的顺序。
- 修改 Prompt 模板。

如果轻微变化导致结论反转，平均分再高也难以部署。

## 多图、长视频和长文档

长上下文评测不仅要增加长度，还要检查检索和归属。

关键问题：

- 模型能否找到相关页、帧或片段？
- 是否把图 A 的信息说成图 B？
- 是否记住早期内容？
- 无关内容增加后，答案是否退化？
- 能否引用正确证据位置？

长上下文能力的核心不是“能塞进去”，而是“能找出来并用正确”。

## Agent 与真实任务评测

Agent 评测关注最终是否完成任务。

| 指标 | 含义 |
| --- | --- |
| Task Success | 任务是否完成 |
| Step Efficiency | 是否使用合理步骤 |
| Recovery Rate | 出错后能否恢复 |
| Tool Accuracy | 工具和参数是否正确 |
| State Tracking | 是否理解环境变化 |
| Cost / Latency | 完成任务的资源成本 |
| Safety | 是否越权或执行危险动作 |

对于 GUI、机器人和工具调用，最终成功率通常比单轮问答分数更重要。

## 多模态安全评测

多模态输入会带来纯文本之外的攻击面：

- 图片中的隐藏指令。
- OCR Prompt Injection。
- 视频帧中的恶意内容。
- 音频中的隐蔽命令。
- 人脸、声纹和位置隐私。
- 伪造图像、视频和声音。
- 工具调用中的越权操作。

安全评测应区分：

1. 模型是否识别风险。
2. 是否拒绝或降级处理。
3. 是否泄露敏感信息。
4. 是否执行了危险动作。
5. 拒答是否误伤正常请求。

只测拒答率不够，还要测帮助性和误拒率。

## LLM-as-a-Judge

LLM Judge 适合评价开放答案，但不是客观裁判。

常见偏差：

- 偏好更长的回答。
- 偏好特定措辞或格式。
- 对自身生成风格更宽容。
- 无法可靠读取高分辨率输入。
- 评分顺序和候选位置影响结果。
- 评判理由听起来合理，但结论不稳定。

提高可信度的方法：

- 明确评分 Rubric。
- 打乱候选顺序。
- 隐去模型名称。
- 使用多个 Judge。
- 用人工标注校准。
- 报告 Judge 与人工的一致率。
- 对高风险样本保留人工复核。

Judge 是测量工具，也需要被评测。

## 人类评测

人工评测适合开放生成、复杂推理和真实体验。

严谨人工评测需要：

- 明确评分标准。
- 对标注员进行培训。
- 隐去模型身份。
- 随机化答案顺序。
- 每个样本由多人评估。
- 报告标注一致性。
- 保留争议样本。

“找几个人看看哪个更好”不是严谨的人类评测。

## 数据污染

公开 Benchmark 很容易进入训练集。

污染可能发生在：

- 原始图片。
- 问题和答案。
- Benchmark 解析文章。
- 图片裁剪或压缩变体。
- 合成数据教师模型。

常见治理方式：

- 文本近似匹配。
- 图像感知哈希。
- 语义检索。
- 时间切分。
- 私有或动态测试集。
- 定期更新样本。

无法排除污染时，应明确报告，而不是把分数直接解释为泛化能力。

## 一套实用评测矩阵

不要只维护一个总榜。建议至少覆盖：

| 维度 | 样例 |
| --- | --- |
| 通用能力 | VQA、Caption、多图问答 |
| 细粒度感知 | OCR、数量、颜色、空间关系 |
| 专项推理 | 图表、数学、科学和文档 |
| 时序与音频 | 视频顺序、时间定位、声音事件 |
| 忠实性 | 幻觉、无答案、证据引用 |
| 鲁棒性 | 改写、压缩、顺序和噪声扰动 |
| 安全 | 隐私、注入、伪造和越权 |
| 系统指标 | 延迟、吞吐、成本和稳定性 |
| 真实任务 | 业务成功率和人工验收 |

公开 Benchmark 用于横向参照，自建评测集用于判断模型是否适合自己的任务。

## 评测与上线闭环

评测不应在发布前临时运行一次。

```text
能力定义
  -> 基准集
  -> 失败样例分类
  -> 数据和训练改进
  -> 回归测试
  -> 灰度上线
  -> 线上监控
  -> 新失败样例回流
```

每次模型、Prompt、视觉编码器、分辨率或推理参数变化，都可能影响结果。

稳定系统需要固定回归集，同时持续补充新样本。

## 如何阅读多模态 Benchmark 论文？

快速看十一件事：

1. 测的是哪种能力？
2. 任务是否必须使用模态输入？
3. 数据从哪里来？
4. 是否排查训练污染？
5. 问题和答案是否存在歧义？
6. 指标是否匹配任务？
7. Prompt 是否统一？
8. Judge 是否经过人工校准？
9. 是否报告分项结果和置信区间？
10. 是否测试鲁棒性与无答案样本？
11. 结论能否外推到真实应用？

如果这些问题没有交代清楚，排行榜名次的意义就很有限。

## 长期不变的判断框架

多年以后，多模态 Benchmark 会不断更新，但下面七个问题不会过时：

1. **能力定义清楚吗？**
2. **题目真的需要使用模态证据吗？**
3. **指标真的测到了目标能力吗？**
4. **数据是否独立、无污染、无明显偏差？**
5. **评分器是否稳定并经过校准？**
6. **模型在扰动、无答案和高风险场景下可靠吗？**
7. **离线分数能预测真实任务成功率吗？**

评测的目的不是产生一个漂亮数字，而是减少我们对模型能力的误判。

## 常见误区

### 1. 综合分高就是全面更强

综合分会掩盖专项短板，应同时看分项结果。

### 2. 回答正确就是看懂了

模型可能依赖语言先验或题目线索，需要做模态消融。

### 3. LLM Judge 比人工更客观

Judge 更便宜、更一致，但同样存在偏差。

### 4. Benchmark 越难越好

难度不是目的。题目应能稳定隔离目标能力。

### 5. 公共榜单可以替代业务评测

业务数据、风险和成功标准通常不同，必须建立自有评测集。

### 6. 模型评测只看准确率

延迟、成本、稳定性、安全和失败恢复同样影响可用性。

## 本章小结

多模态评测的主线可以压缩为：

```text
定义能力
  -> 设计必须使用模态证据的样本
  -> 选择匹配任务的指标
  -> 校准自动评分、Judge 和人工评测
  -> 排除污染与提示偏差
  -> 检查鲁棒性、幻觉和安全
  -> 用真实任务验证离线结论
```

评测不是训练结束后的验收单，而是数据、训练、部署和产品迭代之间的共同语言。
