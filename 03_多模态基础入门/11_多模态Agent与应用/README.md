# 11_多模态 Agent 与应用

> 多模态 Agent 让模型从“理解输入”走向“改变环境”：它观察图像、网页、文档、视频或声音，形成状态判断，调用工具执行动作，再根据反馈继续决策。

**By：猫先生 of 「魔方AI空间」**

## 本章导读

前面的章节主要回答模型如何感知、理解、生成和评测多模态内容。

Agent 进一步回答：

```text
看懂之后，下一步做什么？
```

一个多模态 Agent 的核心闭环是：

```text
观察
  -> 理解当前状态
  -> 规划下一步
  -> 调用工具或执行动作
  -> 获取环境反馈
  -> 验证结果
  -> 继续或结束
```

本章重点回答：

- 多模态模型和多模态 Agent 有什么区别？
- Agent 为什么需要 Grounding、状态、记忆和验证？
- GUI、浏览器、文档、RAG、语音和机器人 Agent 的底层共性是什么？
- 多模态 Agent 最常见的失败发生在哪一环？
- 如何构建多年后仍然适用的 Agent 判断框架？

## 一句话理解多模态 Agent

多模态 Agent 可以理解为：

> 以多模态感知为输入，以环境状态为依据，以工具和动作为输出的闭环决策系统。

它不只是回答：

```text
“这个按钮在右上角。”
```

还要能完成：

```text
定位按钮
  -> 判断是否应该点击
  -> 执行点击
  -> 观察页面变化
  -> 确认任务是否完成
```

从回答问题到完成任务，中间多了行动和反馈。

## VLM 不等于 Agent

VLM 负责理解视觉输入，Agent 负责围绕目标持续决策。

| 维度 | VLM | 多模态 Agent |
| --- | --- | --- |
| 核心任务 | 看懂并回答 | 观察、规划、行动和验证 |
| 输出 | 文本或结构化结果 | 工具调用、点击、输入、移动等动作 |
| 环境 | 通常静态 | 动作后会发生变化 |
| 状态 | 单轮输入为主 | 需要持续维护 |
| 错误处理 | 重新回答 | 需要检测、恢复或回退 |
| 成功标准 | 答案质量 | 任务是否真正完成 |

VLM 是 Agent 的感知组件之一，不是完整 Agent。

## Agent 的七个核心模块

| 模块 | 核心问题 |
| --- | --- |
| 观察 Observation | 当前环境里发生了什么？ |
| Grounding | 语言目标对应哪个对象、区域或时间点？ |
| 状态 State | 当前进度、约束和环境状态是什么？ |
| 规划 Planning | 下一步该做什么？ |
| 工具 Tool | 有哪些可调用能力？ |
| 行动 Action | 如何把意图变成合法动作？ |
| 验证 Verification | 动作是否成功，目标是否完成？ |

这七个模块可以由一个大模型统一完成，也可以由多个专用组件协作。

无论架构如何变化，这个闭环不会消失。

## 观察：不是把所有输入都塞给模型

观察层负责把环境转成 Agent 可用的信息。

输入可能包括：

- 屏幕截图
- DOM / Accessibility Tree
- PDF 页面
- 表格
- 摄像头画面
- 视频帧
- 音频和语音
- 工具执行结果
- 历史动作

观察层的关键不是信息越多越好，而是信息是否足以支持下一步决策。

```text
原始环境
  -> 筛选、压缩和结构化
  -> Agent Observation
```

无关信息过多会增加 Token、延迟和判断噪声。

## Grounding：从语言落到环境

Grounding 是多模态 Agent 的底层难点。

它要把语言中的对象映射到环境中的具体实体：

```text
“点击右上角的保存按钮”
  -> 找到正确按钮
  -> 得到坐标或元素 ID
```

不同环境中的 Grounding：

| 环境 | Grounding 目标 |
| --- | --- |
| GUI | 控件、图标、文本和坐标 |
| 网页 | DOM 节点、链接、表单 |
| 文档 | 页码、段落、表格、单元格 |
| 视频 | 片段、时间戳、人物和事件 |
| 机器人 | 物体、位置、姿态和可达区域 |
| 音频 | 说话人、时间片段和声音事件 |

Agent 经常不是不会规划，而是把正确动作执行在了错误对象上。

## 状态：模型必须知道现在在哪

多步任务需要维护状态。

状态至少包括：

- 用户目标
- 已完成步骤
- 当前页面或环境
- 已获取信息
- 未满足条件
- 工具调用结果
- 错误和重试次数
- 权限与安全约束

如果没有可靠状态，Agent 容易：

- 重复操作。
- 忘记前置条件。
- 在页面变化后继续使用旧坐标。
- 把旧工具结果当成当前事实。
- 提前宣布任务完成。

状态不是聊天记录的同义词。它应该是可更新、可查询的任务表示。

## 记忆：保存什么比保存多久更重要

Agent 记忆可以分为：

| 类型 | 内容 |
| --- | --- |
| 工作记忆 | 当前步骤需要的信息 |
| 情节记忆 | 过去操作和环境反馈 |
| 语义记忆 | 稳定知识、规则和用户偏好 |
| 外部记忆 | 数据库、向量库、文件或任务日志 |

长期记忆不应保存所有原始内容。

更重要的是：

- 哪些信息需要保留？
- 信息何时失效？
- 来源是否可信？
- 新证据是否覆盖旧结论？
- 是否涉及隐私和删除要求？

错误记忆比没有记忆更危险。

## 规划：把目标变成可验证步骤

规划不是把任务写成一长串自然语言，而是把目标拆成可执行、可检查的步骤。

一个好计划应满足：

- 每一步有明确动作。
- 前置条件可判断。
- 结果可以验证。
- 失败后有恢复路径。
- 高风险步骤需要确认。

例如：

```text
目标：提交报销表

1. 打开报销页面
2. 填写金额和类别
3. 上传凭证
4. 检查字段和附件
5. 请求用户确认
6. 提交
7. 验证提交成功
```

规划粒度过粗无法执行，过细则增加成本和错误机会。

## 行动空间

Agent 必须在明确的行动空间中工作。

GUI Agent 的动作可能是：

```text
click(x, y)
type(text)
scroll(direction)
select(option)
wait()
```

工具 Agent 的动作可能是：

```text
search(query)
read_document(id)
query_database(sql)
send_email(to, subject, body)
```

动作接口应具备：

- 明确参数类型。
- 输入校验。
- 权限控制。
- 超时和重试。
- 幂等性设计。
- 可审计日志。

自然语言意图必须经过受约束接口，才能变成可靠行动。

## 工具调用

工具扩展了模型边界。

| 工具 | 弥补的能力 |
| --- | --- |
| OCR | 精确读取文字 |
| 检测 / 分割 | 定位对象和区域 |
| 搜索 / RAG | 获取外部知识 |
| 代码执行 | 计算和数据处理 |
| 浏览器 | 访问网页和完成表单 |
| 数据库 | 查询业务事实 |
| ASR / TTS | 语音输入输出 |
| 机器人控制 | 改变物理环境 |

Agent 的关键不是“会调用更多工具”，而是：

1. 何时调用。
2. 选择哪个工具。
3. 参数是否正确。
4. 如何解释结果。
5. 工具失败后怎么办。

## 反馈与验证

动作执行后必须重新观察。

```text
计划点击“保存”
  -> 执行点击
  -> 页面出现成功提示？
  -> 数据真的写入了吗？
```

只检查“工具没有报错”不够。

验证可以分为：

| 层级 | 检查内容 |
| --- | --- |
| 动作级 | 点击、输入或调用是否成功 |
| 状态级 | 环境是否发生预期变化 |
| 结果级 | 用户目标是否真正完成 |
| 约束级 | 是否违反权限、安全或格式要求 |

没有验证的 Agent，只是在连续猜测。

## 错误恢复

真实环境一定会出错：

- 页面加载失败。
- 控件位置变化。
- 工具超时。
- 权限不足。
- 文件格式错误。
- 用户目标不完整。
- 模型误操作。

恢复策略包括：

- 重新观察。
- 换用结构化接口。
- 回退上一步。
- 重新规划。
- 降级为人工确认。
- 停止高风险动作。

可靠性不取决于永不犯错，而取决于能否发现错误并控制损失。

## GUI Agent

GUI Agent 通过屏幕或界面结构操作软件。

常见输入：

- 截图
- OCR 文本
- DOM
- Accessibility Tree
- 鼠标和键盘状态

常见难点：

- 图标没有文字。
- 页面滚动后坐标变化。
- 弹窗遮挡。
- 相似按钮很多。
- 控件状态只通过颜色表达。
- 动作存在不可逆后果。

GUI Agent 的关键是视觉 Grounding、状态更新和动作验证，不只是截图问答。

## 浏览器 Agent

浏览器 Agent 需要在动态网页中完成搜索、阅读、比较、填写和提交。

典型任务：

- 搜集资料。
- 比较商品或方案。
- 填写表单。
- 预订服务。
- 操作后台系统。

浏览器环境同时包含：

- 页面视觉内容
- DOM 结构
- 网络状态
- 登录和权限
- 异步更新
- 外部不可信文本

因此浏览器 Agent 必须处理 Prompt Injection。网页内容是数据，不应自动被当成系统指令。

## 多模态 RAG

多模态 RAG 不只是把 PDF 做 OCR 后存进向量库。

它需要处理：

```text
文本
图像
页面布局
表格
图表
视频片段
音频片段
```

一个典型流程是：

```text
多模态文档
  -> 解析与切分
  -> 文本、视觉和结构索引
  -> 根据问题检索证据
  -> 多模态模型阅读证据
  -> 生成带引用答案
```

底层关键有三点：

1. **切分**：不能破坏页面、表格和图文关系。
2. **检索**：要找到正确模态和正确粒度。
3. **引用**：答案应能回到页码、区域或时间戳。

## 文档 Agent

文档 Agent 面向合同、报告、论文、票据、表格和知识库。

它需要组合：

- OCR
- 版面理解
- 表格解析
- 跨页检索
- 规则校验
- 结构化抽取
- 证据引用

文档任务中，最危险的不是“不会总结”，而是把错误字段、数字或条款写进后续流程。

高风险文档应保留：

- 原始页面
- 提取位置
- 置信度
- 人工复核入口
- 变更日志

## 语音与视频 Agent

语音 Agent 在实时对话中调用工具；视频 Agent从时间序列中获取状态。

语音 Agent 要处理：

- 流式理解
- 轮次判断
- 用户打断
- 低延迟反馈
- 说话人和情绪

视频 Agent 要处理：

- 抽帧和片段检索
- 动作与事件
- 时间戳
- 长视频记忆
- 连续环境变化

两者共同强调：Agent 的观察是持续变化的，不是一次性输入。

## 机器人与具身 Agent

具身 Agent 把行动扩展到物理世界。

基本闭环仍然相同：

```text
摄像头 / 传感器
  -> 环境状态
  -> 任务规划
  -> 机器人动作
  -> 新观察
```

但物理世界多了约束：

- 动作有成本。
- 错误可能造成损坏。
- 环境部分可观测。
- 控制需要连续性。
- 安全边界更严格。

SayCan、RT-1、PaLM-E、RT-2 等工作体现了语言、视觉和机器人控制逐步结合的路线。

## 应用不应从行业名称开始

医疗、教育、办公、设计、工业只是场景标签。

更稳定的分析方式是先拆任务：

| 任务原型 | 示例 |
| --- | --- |
| 搜索与检索 | 从文档、图片、视频中找证据 |
| 抽取与录入 | 识别字段并写入系统 |
| 比较与审核 | 对照规则发现差异 |
| 规划与执行 | 分解目标并调用工具 |
| 监控与告警 | 观察变化并触发动作 |
| 创作与编辑 | 理解意图并生成内容 |
| 协作与确认 | 与用户多轮澄清和审批 |

行业会变化，任务原型长期稳定。

## 多模态 Agent 的评测

Agent 评测要看完整任务，而不是单轮答案。

| 指标 | 核心问题 |
| --- | --- |
| 任务成功率 | 用户目标是否完成 |
| Grounding 准确率 | 是否找对对象和位置 |
| 动作正确率 | 工具、参数和顺序是否正确 |
| 步骤效率 | 是否有多余或重复操作 |
| 恢复率 | 出错后能否回到正确路径 |
| 验证准确率 | 是否正确判断任务完成 |
| 成本与延迟 | 完成任务耗费多少资源和时间 |
| 安全违规率 | 是否越权、泄露或执行危险动作 |

还要区分：

- 模型规划错误。
- 视觉 Grounding 错误。
- 工具执行错误。
- 环境变化导致失败。
- 评测环境本身不稳定。

只有最终成功率，没有失败归因，很难改进系统。

## 安全与权限

多模态 Agent 能行动，因此风险高于普通问答模型。

核心原则：

### 最小权限

Agent 只获得完成当前任务需要的权限。

### 高风险确认

付款、删除、发布、发送和控制物理设备前，需要明确确认。

### 数据与指令分离

网页、文档、图片和邮件中的内容默认是数据，不能直接提升为系统指令。

### 可审计

记录观察、计划、工具调用、参数、结果和确认过程。

### 可停止和回退

用户应能随时中断，关键操作应支持撤销或补偿。

安全不是最后加一层过滤器，而是行动空间和系统架构的一部分。

## 延迟与成本

多模态 Agent 每一步都可能包含：

- 截图或视频编码
- OCR
- 模型推理
- 检索
- 工具调用
- 新一轮观察

步骤越多，延迟、成本和失败概率越高。

优化重点：

- 减少无效观察。
- 结构化传递状态。
- 对简单任务使用小模型。
- 缓存稳定信息。
- 并行执行独立工具。
- 设置最大步数和预算。
- 在高风险节点保留人工确认。

Agent 的目标不是思考得最多，而是用足够少的步骤可靠完成任务。

## 常见失败模式

### 1. 看对了，点错了

语义理解正确，但 Grounding 或坐标错误。

### 2. 计划正确，状态过期

页面已经变化，Agent 仍按旧观察行动。

### 3. 工具调用成功，任务没有完成

Agent 只检查接口返回，没有验证真实结果。

### 4. 循环和重复操作

没有明确终止条件或状态更新。

### 5. 过度自主

在目标不清、权限不足或风险过高时擅自行动。

### 6. 被外部内容注入

把网页、文档或图片中的恶意文本当成可信指令。

### 7. 记忆污染

错误结果进入长期记忆，影响后续任务。

### 8. Demo 成功，真实环境失败

固定页面和理想数据无法代表动态、噪声和异常场景。

## 工程视角：一个可靠的 Agent 架构

可以把系统拆成：

```text
用户目标
  -> 权限和风险检查
  -> 环境观察
  -> 状态构建
  -> 规划与工具选择
  -> 参数校验
  -> 执行动作
  -> 重新观察
  -> 结果验证
  -> 结束、重试或人工接管
```

建议保留三个独立边界：

1. **模型边界**：模型提出意图，不直接拥有无限权限。
2. **工具边界**：工具接口校验参数、权限和幂等性。
3. **验证边界**：完成状态由证据确认，不由模型一句“已完成”决定。

## 如何阅读多模态 Agent 论文？

快速看十件事：

1. Agent 在什么环境里行动？
2. 观察包含截图、结构数据还是两者都有？
3. Grounding 如何完成？
4. 状态和记忆如何更新？
5. 行动空间是否受约束？
6. 规划、执行和验证是否分离？
7. 工具错误和环境变化如何恢复？
8. 评测看单步准确率还是最终成功率？
9. 是否报告延迟、成本和失败分类？
10. 权限、注入和高风险动作如何控制？

如果只展示成功轨迹，不报告失败和恢复，结论通常不够完整。

## 长期不变的判断框架

多年以后，Agent 框架、工具协议和模型会变化，但下面七个问题不会过时：

1. **看到了正确的环境状态吗？**
2. **语言目标正确落到了对象和位置吗？**
3. **计划能否被拆成合法、可验证的动作？**
4. **每次行动后是否重新观察和更新状态？**
5. **失败时能否发现、回退或请求帮助？**
6. **权限、隐私和高风险操作是否受控？**
7. **系统是否以真实任务成功率为最终标准？**

多模态 Agent 的价值不在于“像人一样说自己会做”，而在于稳定、可控地把事情做完。

## 常见误区

### 1. 接上工具就是 Agent

工具调用只是动作接口。没有状态、反馈和验证，系统仍是单轮调用。

### 2. 模型越大，Agent 越可靠

感知、Grounding、工具、权限和环境稳定性同样决定结果。

### 3. 长计划越详细越好

环境会变化。计划应允许根据新观察调整。

### 4. 使用截图就不需要结构信息

截图提供视觉上下文，DOM、Accessibility Tree 和 API 提供精确结构。两者可以互补。

### 5. Agent 成功一次就说明可用

真实可靠性需要覆盖环境变化、异常、恢复和安全边界。

### 6. 自动化程度越高越好

高风险任务中，合理确认和人工接管是系统能力，不是缺陷。

## 本章小结

多模态 Agent 的主线可以压缩为：

```text
观察环境
  -> Grounding
  -> 更新状态
  -> 规划动作
  -> 受约束地调用工具
  -> 重新观察
  -> 验证结果
  -> 继续、恢复或结束
```

应用场景会变化，但这个闭环长期稳定。

真正可靠的 Agent，不只需要更强模型，还需要清晰状态、受控行动、可验证结果、错误恢复和安全边界。
